# SIP Creator — Business Rules

This document records the business / domain rules enforced by the SIP Creator. These are constraints that come from the application's domain (archiving, e-depot, the Vlaamse Overheid Serieregister contract, …) — not bugs and not implementation details. The aim is so that future readers (us included) can see *that* a rule was a deliberate decision and *why*, rather than guessing whether a check is arbitrary.

When a rule changes, update this file in the same commit.

The reason this file exists is because development with AI is quite standard now, and this file will allow your agent of choice to know why certain decisions were made without having to guess.
Anything not listed here can be treated as non-business rules, so are free to be changed without changing the expected business choices made.

## 1. SIP name

Defined on [SIP](src/utils/data_objects/sip.py) and the dialogs that produce a SIP name.

| Rule | Value / Detail | Source |
| --- | --- | --- |
| Maximum length | 185 characters | [`BusinessRules.SIP_TITLE_MAX_LENGTH`](src/utils/constants.py#L119) |
| May not be empty | Trimmed name must be non-empty | [sip.py:48-55](src/utils/data_objects/sip.py#L48-L55) |
| Forbidden characters | `< > : " / \ \| ? *` | [`SIP.INVALID_NAME_CHARACTERS`](src/utils/data_objects/sip.py#L40) |
| Must be unique per SIP type | A new SIP cannot reuse an existing SIP's name within the same SIP type | [sip.py:71-78](src/utils/data_objects/sip.py#L71-L78) |

**Why 185?** Reserved length for the FTPS file naming pattern `<series_id>-<sip_name>-SIPC.zip`/`.xml` on the e-depot side. Anything longer would risk path-length issues during ingest.

**Why these forbidden characters?** They are the union of characters that are illegal in Windows file names (`< > : " / \ | ? *`). The SIP name is used as a file name (`<name>.db` / `<id>-<name>-SIPC.zip`), so it must be a valid filename on Windows.

## 3. Date validation rules (grid: `Openingsdatum`, `Sluitingsdatum`)

Implemented in [date_check.py](src/utils/grid/checks/common/date_check.py).

These rules apply across all SIP types (digital, migration, analog, onroerend erfgoed) wherever the grid contains an opening- and/or closing-date column.

### 3.1 Format

- Must parse as `YYYY-MM-DD` if non-empty.

### 3.2 Future-date rule

- A date must not be in the future, **except** the year 9999 which is the conventional "open-ended" sentinel and is always allowed.

### 3.3 Series-range rules

- `Openingsdatum` must be **>= series valid_from** and **<= series valid_to**.
- `Sluitingsdatum` must be **>= series valid_from**.
- `Sluitingsdatum` is **NOT** bound by `series valid_to`. A dossier may legitimately be closed off after the series itself was retired.

### 3.4 Pair-ordering

- If both `Openingsdatum` and `Sluitingsdatum` are filled, then `Openingsdatum <= Sluitingsdatum`.

### 3.5 Required-on-dossier

- For rows where `Type == dossier`, both `Openingsdatum` and `Sluitingsdatum` are **required** (must be non-empty).

### 3.6 Pair completeness for non-dossier rows

- If one of (`Openingsdatum`, `Sluitingsdatum`) is filled, the other must also be filled (when the columns are paired in the grid).

### 3.7 Dossier ↔ Stuk hierarchy

- A dossier's `Openingsdatum` must be `<=` the smallest valid `Openingsdatum` of any of its stukken.
- A dossier's `Sluitingsdatum` must be `>=` the largest valid `Sluitingsdatum` of any of its stukken.

## 5. Rijksregisternummer (RRN)

Implemented in [rrn_check.py](src/utils/grid/checks/common/rrn_check.py).

- Accepts either the strict format `xx.xx.xx-xxx.xx` or the loose format `xxxxxxxxxxx` (11 consecutive digits).
- Loose input is auto-formatted to strict form on entry.
- Beyond format, the check digits must satisfy the official Belgian RRN modulo-97 rule (with the post-2000 fallback that prepends `2` to the 9-digit base before the modulo).

## 8. Migration location-column groups

Implemented in [`location_group_check.py`](src/utils/grid/checks/migration/location_group_check.py).

A location group is the four columns: `Origineel Doosnummer`, `Legacy locatie ID`, `Legacy range`, `Verpakkingstype`. Multiple groups are supported by appending spaces to the column names (`"Origineel Doosnummer "`, `"Origineel Doosnummer  "`, …).

- The **first** location group is always required: every row must have all four values filled.
- Subsequent location groups are **all-or-nothing**: a row either fills in all four columns of that group or none of them.
- All four columns of every group must appear together (complete set) and in the canonical order. Incomplete or out-of-order sets cause the overdrachtslijst import to fail with a clear error.

## 9. Re-uploads (REJECTED status)

When the e-depot returns `REJECTED` for an uploaded SIP, the user must be able to re-open the grid, fix the issue, and re-upload. This applies to **all** SIP types (digital, migration, analog, onroerend erfgoed).

A SIP in `REJECTED` state is treated like one that hasn't been uploaded yet: its grid is editable again and its upload action is re-armed.
